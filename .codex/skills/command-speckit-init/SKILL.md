# Spec-Kit: Project Initialization

Bootstrap a `.specify/` directory in the current project to enable spec-driven development.

## Pre-Flight Checks

### Git root detection
!`~/.codex/hooks/speckit-helper.sh check-git-root`

### Existing .specify/ detection
!`~/.codex/hooks/speckit-helper.sh check-specify-dir`

## Instructions

1. **Check pre-flight results above**
   - If `NOT_A_GIT_REPO`: warn the user but allow proceeding (specs still work without git)
   - If `.specify/` `EXISTS`: ask the user whether to reinitialize (overwrite templates only) or abort

3. **Create skeleton `memory/constitution.md`**

   Write this content:
   ```markdown

   <!-- Version: 0.1.0 | Date: YYYY-MM-DD -->

   ## Principles

   2. **[Principle 2]** ŌĆö _Not yet defined._

   _Run `/speckit.constitution` to auto-detect and record._

   ## Architecture Constraints
   _Run `/speckit.constitution` to define._

   ## References
   - Project rules: `.codex/rules/`
   ```

4. **Create `templates/spec.md`**

   Write this content:
   ```markdown
   # Spec: <BRANCH_NAME>
   <!-- Auto-generated by /speckit.specify -->

   ## Overview
   _Brief description of the feature._

   ## User Scenarios

   ### US1: <Scenario Name> [P1]
   - **Given** <precondition>
   - **When** <action>
   - **Then** <expected outcome>

   ## Functional Requirements

   | ID | Requirement | Priority | Scenario |
   |----|-------------|----------|----------|
   | FR-001 | _description_ | P1 | US1 |

   ## Success Criteria

   | ID | Criterion | Validation Method |
   |----|-----------|-------------------|
   | SC-001 | _description_ | _how to verify_ |

   ## Clarifications
   _None yet._
   ```

5. **Create `templates/plan.md`**

   Write this content:
   ```markdown
   # Plan: <BRANCH_NAME>
   <!-- Auto-generated by /speckit.plan -->

   ## Research Notes
   _Resolved clarifications and technical findings._

   ## Data Model
   _Key entities and relationships. See `data-model.md` for full schema if needed._

   ## API Contracts
   _Endpoint signatures, request/response shapes. See `contracts/` for details if needed._

   ## Implementation Approach
   _High-level strategy, key decisions, trade-offs._

   ## Constitution Compliance
   - [ ] Principle 1: _how this plan aligns_
   - [ ] Principle 2: _how this plan aligns_

   ## Quick Start
   _Steps to begin implementation after plan approval._
   ```

6. **Create `templates/tasks.md`**

   Write this content:
   ```markdown
   # Tasks: <BRANCH_NAME>
   <!-- Auto-generated by /speckit.tasks -->

   ## Phase 1: Setup
   - [ ] T001 [US1] _Setup task description_ `path/to/file`

   ## Phase 2: Foundational
   - [ ] T002 [US1] _Foundation task_ `path/to/file`

   ## Phase 3: User Stories (by priority)
   ### P1
   - [ ] T003 [P] [US1] _Implementation task_ `path/to/file`

   ### P2
   - [ ] T004 [US2] _Implementation task_ `path/to/file`

   ## Phase 4: Polish
   - [ ] T005 [US1] _Polish task_ `path/to/file`

   ---
   Legend: `[P]` = parallelizable | `[US#]` = user scenario ref
   ```

7. **Create `templates/checklist.md`**

   Write this content:
   ```markdown
   # Checklist: <TYPE>
   <!-- Auto-generated by /speckit.checklist -->

   | ID | Check | Dimension | Status |

   | CHK002 | _check description_ | [consistency] | [ ] |
   | CHK003 | _check description_ | [testability] | [ ] |
   ```

8. **Post-init guidance**
